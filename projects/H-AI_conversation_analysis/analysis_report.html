5. 개선 권고 (RAG·프롬프트·피드백 루프)

아래 '개선_권고'는 분석 결과를 바탕으로 한 자동 생성 요약입니다. **구체적인 RAG 강화 항목·프롬프트 수정안**은 Py 로직으로 생성하지 않으며, 분석 JSON(analysis_report.json)이 생성된 뒤, 채팅에서 '이 JSON 기반으로 RAG·프롬프트 구체 개선안을 LLM으로 종합 분석해 달라'고 요청한 후, 생성된 내용을 JSON의 **'개선_권고_상세'** 항목에 수동으로 붙여넣는 방식을 권장합니다. JSON 구조는 '개선_권고_상세' 아래에 rag_강화_상세, 프롬프트_튜닝_상세, 피드백_루프_상세 등으로 넣으면 HTML 보고서에서 함께 노출할 수 있습니다.

프롬프트 튜닝 상세

  • 직접 답변 우선(첫 문장 규칙)
    질문이 Yes/No·가능 여부·의미·개념을 묻는 경우, 답변 첫 문장에 반드시 그에 대한 직접 답을 포함하라. 예: '일정 변경은 가능합니다. 단, …' / '예약번호는 예약마다 부여되는 고유 번호를 의미합니다. 확인하시려면 …'오답 사유에 '질문에 대한 직접적인 답변 부재', '~만 제공하고'가 다수. 질문 의도(가능 여부·의미·차이)에 맞는 한 문장을 먼저 제시한 뒤 확인 방법·부가 설명을 이어가도록 시스템 지시에 명시.
  • 의도 불명확 시 선택지
    질문이 한 단어, 날짜만, URL만이거나 문맥이 부족하면: '다음 중 어떤 문의인가요? (1) … (2) … (3) 그 외(한 줄만 더 적어주세요)' 형태로 선택지를 제시한 뒤, 사용자 응답에 따라 답변하라.기타문의·욕설인사비문 오답에서 '출발일 9월 둘째주', '3월', '옷은 어떻게' 등에 엉뚱한 상품·일반론만 제공. 의도 확인을 위한 선택지 템플릿을 프롬프트에 포함.
  • 용어/개념 질문 시 개념 선행
    예약번호, 발권, 예약대기, 세이브 스탠다드 등 '~가 뭐예요?' 질문에는 먼저 '~는 …를 의미합니다'로 개념을 한 문장 설명한 뒤, '확인하시려면 …' 등 방법을 안내하라. '확인 방법'만 말하지 말 것.하나투어 이용(FAQ) 오답에서 '의미'를 물었는데 '확인 방법'만 제공하는 패턴 반복. RAG 용어 FAQ와 프롬프트 규칙을 맞춰 개념 설명을 우선하도록.
  • 일정·변경·가능 여부 질문 시 선행 답
    '변경 가능한가요?', '취소 가능한가요?'에는 먼저 '가능합니다' 또는 '불가능합니다'(또는 '조건에 따라 다릅니다')를 밝힌 뒤, 절차·조건을 설명하라. 일정 제안·추가 정보 요청만으로 대체하지 말 것.샘플에서 '일정 변경 가능한가요?'에 마쓰야마 일정 제안만 한 사례 다수. 가능 여부를 먼저 답하는 규칙을 프롬프트에 추가.
  • 상품 비교 질문 시 구체 비교 또는 정보 요청
    두 상품/가격 차이를 물으면: 검색 결과에 해당 상품이 있으면 가격·포함 내역·일정을 항목별로 나열해 비교하고, 없으면 '비교할 상품 번호나 링크를 알려주시면 비교해 드리겠습니다'라고 한 문장으로 답하라. 일반적인 여행 상담 말투로 대체하지 말 것.179,900원 vs 189,000원 차이에 일반론만 제공하는 등 상품 비교 오답 감소를 위한 규칙.

피드백 루프 상세

  • 오답 유형 태깅 및 RAG/프롬프트 매핑
    오답사유를 '의미 vs 방법 혼동', '직접 답 부재', '조건 미반영(상품 검색)', '문맥 미반영(짧은 질문)' 등 태그로 분류해, 주기적으로 어떤 RAG 문서·프롬프트 규칙을 보강할지 우선순위에 반영. 본 분석의 오답_원인_분류·카테고리별_오답를 태깅 스키마 참고로 사용.
  • 오답사유 빈도 구문 주기 점검
    오답사유_빈도_구문에서 '만 제공하고', '에 대한 정보' 등 반복 구문이 새로 늘어나면 '직접 답 대신 다른 정보만 제공' 패턴이 증가한 것으로 보고, 해당 기간 오답 샘플로 프롬프트·RAG를 재점검.
  • 의도 클러스터와 오답 카테고리 교차 분석
